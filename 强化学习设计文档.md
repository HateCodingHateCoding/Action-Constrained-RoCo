# 多机械臂协作 + 强化学习扩展设计文档

> 项目：RoCo 复现与扩展（智能系统期末项目）
> 目标：用强化学习屏蔽 LLM 幻觉，把 LLM 的"开放式问答"变成"选择题"

---

## 0. 为什么要加 RL

RoCo 让多个机械臂通过 LLM 对话协作完成任务。问题在于 LLM 会"幻觉"——
比如让 Alice 去拿一个她根本够不到的方块，整个计划就崩了。

我们的扩展思路是：**把 LLM 的"开放式问答"变成"选择题"**。
做法是在 LLM 之上加一个强化学习（RL）智能体作为"高层动作选择器"：

- LLM 仍然负责理解任务、提出候选动作（带语义解释）；
- RL 在**合法动作集合**里挑最终动作，能在 LLM 不靠谱时纠偏；
- 底层 IK + RRT 路径规划保持不变。

一句话类比：LLM 像"满嘴跑火车的项目经理"，RL 像"老练的执行总监"——
经理列几个方案，总监挑出真正能落地的那一条。

## 1. 整体架构

```
+---------------------------------------------+
|  MuJoCo 环境 EnvState                       |
|  机械臂位姿 / 物体位置 / 接触关系           |
+---------------------+-----------------------+
                      | obs
                      v
+----------------+   +-----------------------+
| LLM 对话       |-->| RL 高层策略 π(a|s)    |
| (候选 k 条)    |   | + 动作合法性 mask     |
+----------------+   +----------+------------+
                                | a=(obj,target)
                                v
                     +-----------------------+
                     | 底层执行: IK + RRT    |
                     | -> SimAction->step()  |
                     +-----------------------+
```

要点：RL 决定"选哪个 PICK·PLACE"，属于 option-level（高层离散）策略；
不直接控制关节力矩，sample efficiency 高、训练稳。

## 2. 动作空间 Action Space

### 2.1 直观理解

每个机械臂在每一步只做一个高层决策——"拿哪个东西、放到哪里"，或者"原地等"。
也就是说：

```
动作 = WAIT  或  PICK(物体X) → PLACE(目标Y)
```

例如 sort 任务里 Alice 的合法动作可能有：
- `WAIT`
- `PICK(blue_square) PLACE(panel2)`
- `PICK(yellow_trapezoid) PLACE(panel3)` ← 帮 Chad 中转
- ……

### 2.2 数学形式

智能体 i ∈ {Alice, Bob, Chad}，每步动作：

```
a_i ∈ {WAIT} ∪ { (o, t) | o ∈ 可抓物体_i,  t ∈ 可放位置_i }
```

联合动作 a = (a_1, ..., a_N)（CTDE：训练集中、执行分布式）。

### 2.3 编码：两个 head 的因子化输出

如果直接把 (o,t) 当一个大类别，类别数随物体/目标数爆炸。
所以拆成两个独立的 softmax head：

| Head        | 含义              | sort 任务下维度 |
| ----------- | --------------- | ---------- |
| `head_obj`  | 选哪个物体（含 "WAIT"） | 4          |
| `head_tgt`  | 放到哪个目标          | 7          |

最终 logits = `softmax(head_obj) ⊗ softmax(head_tgt)`。

### 2.4 动作 mask（这是把"问答"变"选择题"的核心）

每一步根据当前状态生成一个布尔 mask，把这些位置 logit 设成 −∞：

- 物体不在我臂展范围内 → 不能 PICK
- 我手上已经抓着东西 → 不能再 PICK
- 目标 panel 在我臂展外 → 不能 PLACE
- 任务规则禁止的搭配（比如 blue_square 只能放 panel2/3/5）

mask 的合法性逻辑直接复用 [task_sort.py:482](rocobench/envs/task_sort.py#L482) `get_task_feedback` 里已经写好的检查。

### 2.5 与 LLM 的耦合（推荐方案）

```
LLM 给 top-k 候选 ─┐
                   ├─→ 合法 mask ∩ LLM 候选 ─→ RL argmax
合法性约束 ─────────┘
```

实现上：把 LLM 没提到的动作 logit 减一个学习得到的 β，让 RL 在 LLM 靠谱时跟随、
幻觉时纠偏。这就是项目里"做选择题"的具象化。

### 2.6 跨任务通用性

sort / pack / sweep / sandwich / cabinet / rope 都能映射到 "动词 + 名词" 模板：

| 任务         | 主要动词                       |
| ---------- | -------------------------- |
| sort       | PICK, PLACE, WAIT          |
| pack       | PICK, PLACE, WAIT          |
| sweep      | PICK, PLACE, SWEEP, DUMP   |
| sandwich   | PICK, PLACE, WAIT          |
| cabinet    | OPEN, PICK, PLACE          |
| rope       | PICK, PLACE                |

每个任务的合法动词列表写在它自己的 task class 里。

## 3. 状态空间 State Space

### 3.1 直观理解

智能体每一步要"看清楚环境"，需要的信息无非三类：

1. **我自己**：手臂在哪、抓着没、抓的是啥；
2. **物体**：每个方块在哪、谁拿着；
3. **任务进展**：哪些子目标已经完成、还差什么。

把这些信息归一化、拼成一个长向量喂给神经网络即可。

### 3.2 智能体自身特征（约 30 维 / agent）

直接从 [base_env.py:58-93](rocobench/envs/base_env.py#L58-L93) 的 `RobotState` 抽：

| 字段               | 维度       | 说明                          |
| ---------------- | -------- | --------------------------- |
| 末端位置 ee_xpos     | 3        | 归一化到 SCENE_BOUNDS 内 [-1, 1] |
| 末端朝向 ee_xquat    | 4        | 四元数                         |
| 关节角 qpos         | 6×2 = 12 | 用 (sinθ, cosθ) 编码避免角度跳变     |
| 关节速度 qvel        | 6        |                             |
| 是否在抓东西 is_grasp  | 1        | `len(contacts) > 0`         |
| 抓的物体 onehot      | \|O\|    | 在哪个物体上                      |
| 自身可达 bbox        | 6        | get_robot_reach_range 的边界   |

### 3.3 物体特征（每物体约 15 维）

| 字段                    | 维度       | 说明                           |
| --------------------- | -------- | ---------------------------- |
| 位置 xpos / 朝向 xquat    | 3 + 4    | 归一化                          |
| top/bottom 高度         | 2        |                              |
| 当前在哪个 panel 上 onehot  | \|panel\| | 来自 `get_cube_panel`          |
| 谁拿着 onehot            | N + 1    | 在某 agent 手上 / 在桌面            |
| 是否是"我"的目标             | 1        | 用于让 actor 关注自己的活儿            |
| 各 agent 是否够得到         | N        | 用于协作判断                       |

### 3.4 任务 / 协作特征（共享，约 10 维）

- 子目标完成度 onehot（如 sort：blue_in_p2、pink_in_p4、yellow_in_p6）；
- 上一时刻每个 agent 的动作（让策略稳定，避免反复横跳）；
- 全局 timestep（归一化）；
- **当前合法动作 mask 也是观测的一部分**——critic 训练时可以用，actor 用来限制采样。

### 3.5 多智能体范式：CTDE

- **训练时**：critic 看全局拼接观测（所有 agent 的局部观测 + 物体 + 任务进度）；
- **执行时**：每个 actor 只看自己的局部观测 + mask。

这与 RoCo 的 "对话后各自执行" 语义天然对齐。

## 4. 奖励函数 Reward Function

奖励分三层：**任务奖励**（你做对了吗）+ **塑形奖励**（你接近目标了吗）+ **协作/约束奖励**（你的姿势对吗）。

### 4.1 任务奖励（稀疏，正确性）

```
R_goal_i  = +10   首次把物体 o_i 放到正确目标位
R_done    = +50   全部子目标完成（沿用 task_sort.get_reward_done 的判断）
```

只在"首次满足"时给，避免重复刷分。

### 4.2 进度塑形奖励（稠密，效率）

用 **potential-based shaping**（Ng et al. 1999），理论上不会改变最优策略：

```
Φ(s) = − Σ_i w_i · ‖物体_i 到目标 panel 的水平距离‖
R_shape(s, s') = γ · Φ(s') − Φ(s)
```

直观上：物体离目标越近 → Φ 越大 → 一步走得更近就给一点正奖励。

### 4.3 协作奖励（这是 RoCo 扩展的灵魂）

| 信号           | 触发条件                                            | 数值      |
| ------------ | ----------------------------------------------- | ------- |
| `R_handoff`  | 把"不属于自己、原 agent 又够不到原位"的物体，挪到下一个 agent 的可达范围 | **+2** |
| `R_block`    | 占用了一个 panel，导致别人放不下他的目标物                       | **−1** |

这两项直接对应任务上下文："如果你的方块够不到，请别人帮忙"。

### 4.4 约束 / 惩罚（把"幻觉"变成训练信号）

| 信号                  | 触发条件                  | 数值      |
| ------------------- | --------------------- | ------- |
| `R_invalid_pick`    | 选了不可达物体 / 已抓着东西还 PICK | **−2** |
| `R_invalid_place`   | 选了不可达 panel           | **−2** |
| `R_collision`       | RRT 规划失败 / 碰撞         | **−1** |
| `R_all_wait`        | 所有 agent 同时 WAIT      | **−0.5** |
| `R_step`            | 每步恒定（鼓励尽快完成）          | **−0.05** |
| `R_drop`            | 物体掉到桌外 / 出 SCENE_BOUNDS | **−5**（并 done） |

> 注：mask 已经会屏蔽掉非法动作，但保留惩罚是双保险——
> 万一 mask 写漏了一种情况，惩罚信号会把策略"拉回来"。

### 4.5 总奖励

```
R_total = Σ_i [R_goal_i + R_handoff_i]
        + R_done + R_shape
        + Σ_i [R_invalid_pick + R_invalid_place + R_collision]
        + R_all_wait + R_step + R_drop
```

## 6. 训练算法选择

推荐 **MAPPO**（多智能体 PPO）：

- on-policy、稳定、超参好调；
- 动作离散且有 mask，正好用 `MaskedCategorical` 分布；
- CTDE 友好：共享一个全局 critic，每个 agent 一个独立 actor。

备选 **QMIX**：value-based、sample efficiency 高，但 mask 实现稍复杂。

训练流程建议：
1. 先在 **sort 任务**单独跑通（任务最简单，3 个 agent，状态干净）；
2. 跑通后冻结超参，迁移到 sweep / pack；
3. 最后做对比实验：纯 LLM vs 纯 RL vs LLM+RL。

## 7. 实验设计建议

### 7.1 对比实验

| 方法                | 说明                  |
| ----------------- | ------------------- |
| **LLM-only**      | RoCo 原版（baseline）   |
| **RL-only**       | 完全替代 LLM 高层规划       |
| **LLM + RL（我们的）** | LLM 候选 + RL mask/选择 |

指标：成功率、平均步数、协作动作占比、动作合法率。

### 7.2 消融实验

- 去掉 mask（看 RL 能不能自己学到合法约束）；
- 去掉 shaping reward（看任务奖励够不够稀疏地学出来）；
- 去掉 handoff reward（看会不会退化为各干各的）；
- 改变 LLM 候选 k 的数量（k=1, 3, 5, 全开放）。

## 8. 一句话总结

**状态空间**告诉网络"现在长什么样"，
**动作空间**约束网络"只能选哪些"，
**奖励函数**告诉网络"这样选对不对"。

三者搭在 RoCo 现有的 IK + RRT + LLM 框架上，就把"靠 LLM 自由发挥"变成了
"在合法集合里做选择题"——这正是我们扩展的核心价值。

## 9. 实验结果与发现（实跑数据）

### 9.1 符号化 Sort 环境（200 局，sort 任务）

| 方法 | 成功率 | 平均步数 | 非法动作 | 备注 |
| --- | ---: | ---: | ---: | --- |
| random_no_mask | 1.0% | 29.9 | 80.2 | 无约束基本不可解 |
| random_masked | 65.5% | 22.1 | 7.5 | 仅 mask 已经把成功率拉到 65% |
| scripted | 22.0% | 24.3 | 0.0 | 写死规则解不了多跳协作 |
| **mappo_full（我们的）** | **100%** | **3.6** | 0.3 | RL+mask+塑形+协作奖励 |
| mappo_no_mask（消融） | **1.5%** | 29.8 | 0.1 | **训练时无约束 → 学到作弊策略** |
| mappo_no_shaping | 100% | 3.8 | 0.2 | 只影响收敛速度，不影响终态 |
| mappo_no_handoff | 100% | 3.6 | 0.1 | 同上 |
| **hybrid (LLM+RL)** | 90% | 7.1 | 0.5 | LLM 候选 + 反幻觉守卫 + RL 选择 |

### 9.2 真实 MuJoCo Sort 环境（5 seed）

| 方法 | 成功率 | 平均步数 |
| --- | ---: | ---: |
| LLM-only (glm-4-flash, dialog 模式) | **0% (0/3)** | 8 (timeout) |
| **mappo (我们的)** + `--snap-finished` | **60% (3/5)** | 5.6 |

### 9.3 三个值得讲的发现

1. **mask 是核心，不是优化项**（mappo_no_mask 1.5%）。训练时去掉 mask，
   RL 学到的是"在没有约束的训练环境里作弊"的策略，一旦上线被合法性约束
   屏蔽 → 直接崩。这正面验证了"做选择题"的核心假设。

2. **LLM 不靠谱时硬拉它会拉低性能**（hybrid 早期版本 0% → 加反幻觉守卫
   后 90%）。具体表现：glm-4-flash 倾向"全 WAIT"，硬当真就让 RL 也躺平。
   守卫规则一行：**LLM 全 WAIT 就忽略候选**，立刻回升到 90%。
   这是个很好的"工程上 LLM 不能盲信"的实证。

3. **scripted 写不了多跳协作**（22%）。手写规则只能处理"我能直接拿到
   就拿"的情形，处理不了"梯形方块在 panel2，需要 Alice → Bob → Chad
   三跳"这种长链协作。**协作策略必须学出来**。

### 9.4 失败模式（诚实的局限）

- LLM-only 0/3：glm-4-flash 反复把 pink_polygon 放到 panel2（应该是 panel4），
  即使 env feedback 说"不合法"也学不会。这正是项目要解决的"幻觉"。
- 真实 MuJoCo 失败的 2 个 seed：底层 RRT 在某些多臂位姿下找不到无碰撞路径，
  属于"复现+迁移"组的 IK 调优范围，不影响策略本身。

## 10. 跨任务通用性

`sweep` 任务的 vocab/mask/奖励钩子已经做完（[task_sweep.py](rocobench/envs/task_sweep.py)），
对应的符号化训练环境也写好了（[sweep_symbolic_env.py](rocobench/rl/sweep_symbolic_env.py)）：

| 维度 | sweep 任务 | sort 任务 |
| --- | --- | --- |
| 智能体 | Alice, Bob | Alice, Bob, Chad |
| 动词 | WAIT/MOVE/SWEEP/DUMP | WAIT/PICK·PLACE |
| 目标 | cube×3, trash_bin, self | panel×7 |
| mask 关键规则 | SWEEP 仅 Bob、DUMP 仅 Alice 且簸箕非空、SWEEP 需 Alice 同步 MOVE | reach 区间 + 物体-面板对应 |

**实跑验证**：30k 步训练，**评估 50 局成功率 100%，平均 7.24 步**。

**核心设计验证**：换任务时 RL 训练代码、actor/critic 网络、训练循环
**一行没改**。只填了 vocab 和 mask 两个 hook，加 SweepSymbolicEnv 这一个文件。
这就是"框架统一、词表分任务、mask 按任务写规则"的实证。

学到的策略还自发发现了**流水线**：t=2 时 Alice 已经在 dump，
Bob 已经在去下一个方块的路上——比手写顺序节省 2 步。

## 11. Mask-Aware LLM 提示消融

让 LLM 在生成动作时**也看到合法 mask**。结果（GLM-4-flash, sort 任务 8 局）：

| 提示策略 | 成功率 | 平均步数 | LLM 非法动作/步 |
| --- | ---: | ---: | ---: |
| vanilla (LLM 只看任务说明) | 62.5% | 14.5 | **1.67** |
| **mask-aware** (LLM 也看到合法动作集) | **100%** | **4.25** | **0.35** |

**结论**：把 mask 喂给 LLM，LLM 的幻觉率下降 **4.7×**，混合策略成功率
从 62.5% → 100%。这佐证了一个更深的论点——**合法性约束是可共享的辅助信息**：

- RL 训练时用 mask 屏蔽非法动作；
- LLM 生成时用 mask 缩小输出空间；
- 二者共用一套约束规则，不需要额外标注或人工调参。

这把"做选择题"的范式从"训练 RL 选"扩展到"也教 LLM 怎么选"——
**约束本身就是协议**。
